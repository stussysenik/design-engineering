# VISION

As of April 1, 2026.

## North Star

Become the engineer people call when a product is:

- technically complex
- operationally messy
- interaction-heavy
- visually underpowered
- important enough to deserve taste

The lane is:

`high-leverage product systems with exceptional experience quality`

Not “I know every language.”
Not “I can do a little of everything.”

## The One Thing To Double Down On

Double down on:

`interaction systems for complex tools`

That includes:

- information architecture
- state transitions
- motion with purpose
- trust and visibility
- keyboard fluency
- latency handling
- composable primitives
- operational clarity

This is rare, portable, and senior.

It bridges:

- web apps
- iOS
- CLI/TUI
- internal tools
- AI products
- workflow software

## The Current Wedge

The current flagship is:

`toolmotion`

This is the upstream system.
It proves your point of view about how serious software should feel.

## What Toolmotion Should Become

`toolmotion` should become:

1. a motion system
2. a pattern library for serious software
3. a demo app proving the patterns
4. a React adapter
5. eventually a cross-platform reference point

It is not just an animation package.

It should answer:

`What should motion feel like in serious software?`

## Product Sequence

### Project 1

`toolmotion`

Build the system first.
Make the semantics solid.
Prove them in real tool surfaces.

### Project 2

`AI-native iOS utility`

Build one strong Apple-native product that turns messy input into structured action.

Good directions:

- screenshot to task/issue workflow
- camera/document review tool
- local or hybrid inference utility

### Project 3

`Operator`

A workflow and artifact interface for AI and engineering teams.

This is still a good downstream product, but it should come after `toolmotion` gives you a reusable interaction language.

## Why This Direction Matters

This path is stronger than more one-off demos because it creates:

- a reusable system
- a public point of view
- a community-facing artifact
- a bridge into companies like Windmill, Vercel, Linear, and OpenAI

## Windmill Fit

If you want to be compelling for Windmill, your work should prove:

- local-dev empathy
- CLI taste
- workflow thinking
- strong frontend craft
- ability to make internal software feel first-class
- strong code and interaction judgment in the same person

The first proof is not another landing page.
It is a reusable interaction system that shows how serious tools should behave.

The second proof can be a real workflow product built on top of that understanding.

## What To Learn Deeply

You do not need to build every layer from scratch.
You do need enough first-principles understanding to make strong design and engineering decisions.

The best pattern is:

- choose a real vertical problem
- understand the relevant layers deeply
- ship the product-level abstraction

That is better than broad “from scratch” tourism.

## What To Avoid

- gradients as identity
- endless public experiments
- language collecting as public narrative
- clones as flagship work
- framework churn on the critical path

## 12-Month Goal

By this time next year, the public story should be:

- one flagship open-source interaction system people actually use
- one high-quality Apple-native app
- one strong essay series tied to shipped artifacts
- one clear role identity

The account should read:

`serious design engineer with unusual systems depth`

not:

`interesting builder with many experiments`
